# Codex 5.6 models + Sol-as-judge review subsystem — Design

**Date:** 2026-07-10
**Status:** Draft (brainstorming output, pending spec review)
**Supersedes assumptions in:** `templates/model-policy.toml` (flagship-first),
`skills/codex-delegate/SKILL.md` ("you choose effort, not a model")

---

## 1. Context & motivation

OpenAI shipped the **GPT-5.6** line — three sibling models with different
power/cost profiles:

- **sol** — frontier agentic coding model (strongest; most expensive).
- **terra** — the everyday workhorse (~ parity with a top general model).
- **luna** — the small/cheap sibling.

This changes two things the current plugin got right for a single-flagship world
but that no longer fit:

1. **Model choice is now a lever, not a constant.** Today the policy is
   "flagship-first": one model (`gpt-5.5`) for every class, and the task class
   only picks *effort*. With three tiers of genuinely different cost/precision,
   the model itself should vary by task.
2. **Codex can now exceed Claude's precision on judgment tasks.** sol in
   particular is strong enough that its review of code/plans is a real second
   opinion, not just a mechanical executor's output. The plugin should expose
   **read-only "judge" capabilities** (code-review, audit, plan-review) driven
   by the strongest model as an additional correctness guarantee — while Claude
   still critically evaluates every finding rather than deferring blindly.

### 1.1 Ground truth (verified 2026-07-10, not assumed)

The model IDs were **verified against the installed CLI**, not taken on faith —
the existing template already warns that an ID the account cannot run "burns all
its fallback attempts and hands back with no work done."

Verification path and findings:

- The account's `~/.codex/models_cache.json` under CLI **0.142.5** listed only
  `gpt-5.5`, `gpt-5.4`, `gpt-5.4-mini`, `codex-auto-review` — **no 5.6**.
- `~/.codex/version.json` claimed `0.142.5` was "latest" (checked 2026-07-06),
  but that record was stale. `codex update` moved the CLI to **0.144.1**.
- Under 0.144.1, `codex debug models` (renders the raw model catalog as JSON)
  revealed the 5.6 line. **Real slugs and effort ladders:**

  | slug (`-m` value)  | display        | supported efforts                         | api |
  | ------------------ | -------------- | ----------------------------------------- | --- |
  | `gpt-5.6-sol`      | GPT-5.6-Sol    | low·medium·high·xhigh·**max·ultra**       | yes |
  | `gpt-5.6-terra`    | GPT-5.6-Terra  | low·medium·high·xhigh·**max·ultra**       | yes |
  | `gpt-5.6-luna`     | GPT-5.6-Luna   | low·medium·high·xhigh·**max** (no `ultra`) | yes |
  | `gpt-5.5`          | GPT-5.5        | low·medium·high·xhigh                      | yes |
  | `gpt-5.4`          | GPT-5.4        | low·medium·high·xhigh                      | yes |
  | `gpt-5.4-mini`     | GPT-5.4-Mini   | low·medium·high·xhigh                      | yes |
  | `codex-auto-review`| (hidden)       | (used by native `codex review`)           | yes |

- The CLI gained a **native `codex review`** subcommand
  (`--uncommitted`, `--base <branch>`, `--commit <sha>`, prompt via stdin) backed
  by the dedicated `codex-auto-review` model. This is best-in-class for diff
  review and should be reused rather than re-implemented.

**Lesson baked into the design:** a stale CLI can *hide* newer models while
`version.json` still reports itself "latest." Doctor must therefore check the CLI
version freshness, not just presence.

### 1.2 One residual verification (runtime, ChatGPT-auth)

The catalog marks every 5.6 model `supported_in_api: true`, but this account uses
**ChatGPT auth** (`auth.json`, `cli_auth_credentials_store = "file"`), and the
template's known footgun is that some catalog models are still rejected on
ChatGPT accounts with *"model is not supported when using Codex with a ChatGPT
account."* The design does **not** assume runtime availability — the new doctor
model-availability probe (§4) is exactly what confirms each configured slug is
runnable on the active account. The implementation plan includes a one-time probe
of `gpt-5.6-{sol,terra,luna}` before the new template is committed as default.

## 2. Goals / non-goals

### Goals

1. **Per-class model + effort.** Task classes select both a model and an effort;
   the model tier is a first-class cost/precision lever.
2. **Adjust the effort ladder to the real catalog** — precisely: **add `max` and
   `ultra`, remove `minimal`** (`low`/`medium`/`high`/`xhigh` already exist
   today). Then **validate `(model, effort)` against the live catalog** so an
   unsupported pair (e.g. `luna + ultra`) fails at load/doctor time, never
   mid-run.
3. **A read-only "judge" subsystem**: `code-review`, `audit`, `plan-review`,
   with sol as the correctness guarantee, exposed so Claude is fully aware of
   them at planning time.
4. **Harden doctor**: model-availability probe + CLI-version freshness; make
   `refresh-models` actually work via `codex debug models`.
5. **Rewrite `SKILL.md` into a capability map** — the primary deliverable — so
   that when Claude plans and is told "you may delegate to Codex here," it knows
   the full arsenal (execute vs judge, which model, which effort, the cost
   discipline, and the critical-evaluation contract).

### Non-goals

- Automating model *pricing* logic or budgets/hard cost gates. Cost discipline is
  guidance in the skill; Claude stays autonomous (explicit user requirement).
- Making reviews a blocking gate. Reviews are **advisory** — Claude evaluates
  findings critically and decides. (The existing `checks` mechanism remains the
  only hard, blocking gate, and only for the *execute* path.)
- Multi-turn/interactive review conversations. Reviews run non-interactively like
  the rest of the tool.
- Changing the execution safety contract (whitelist, protected-path deny-list,
  clean-tree preflight, `approval_policy="never"`). Those are untouched.

## 3. Model policy: per-class model + effort

### 3.1 Type changes (`src/config/types.ts`)

- `Effort` today is `'minimal' | 'low' | 'medium' | 'high' | 'xhigh'`
  (`types.ts:1`). The change is **additive + one removal**, not a wholesale
  replacement: add `'max'` and `'ultra'`, remove `'minimal'`.
  ```ts
  export type Effort = 'low' | 'medium' | 'high' | 'xhigh' | 'max' | 'ultra';
  ```
  Removing `'minimal'` is a breaking change to an *exported* type, so the plan
  must first grep the codebase + templates + tests and confirm nothing resolves
  to `'minimal'` (the shipped template uses only `low`/`medium`/`high`; the
  fixture policy likewise). If any local override uses it, doctor's `models` row
  will flag it after the change.
- **`ModelTier` is kept as informational metadata, not dropped.** `types.ts:2`
  defines `ModelTier = 'flagship' | 'fast' | 'general'`; `ModelEntry` still
  carries `tier`. The 5.6 slugs map onto the existing values:
  `gpt-5.6-sol → flagship`, `gpt-5.6-terra → general`, `gpt-5.6-luna → fast`
  (and the 5.4/5.5 fallbacks keep their current tiers). `tier` stays purely
  descriptive — nothing in `resolve()` branches on it — so no `ModelTier` value
  changes are needed.
- `ModelEntry` gains the catalog effort set, so the policy loader can validate
  pairs without a live network call every time:
  ```ts
  export interface ModelEntry {
    readonly tier: ModelTier;
    readonly efforts?: readonly Effort[]; // supported reasoning levels; optional for back-compat (§8)
  }
  ```
- A new `ReviewClassConfig` (same shape as `TaskClassConfig`) + a `[review]`
  section on `ModelPolicy` (see §5.2).

### 3.2 Template rewrite (`templates/model-policy.toml`)

Concrete v1 mapping (grounded in the verified catalog):

| use                          | model           | effort  | fallback (availability only) |
| ---------------------------- | --------------- | ------- | ---------------------------- |
| `mechanical`                 | `gpt-5.6-luna`  | medium  | `gpt-5.4-mini`, `gpt-5.4`    |
| `implementation` *(default)* | `gpt-5.6-terra` | medium  | `gpt-5.5`, `gpt-5.4`         |
| `hard`                       | `gpt-5.6-sol`   | high    | `gpt-5.6-terra`, `gpt-5.5`   |

Design rules encoded as template comments:

- **Never minimize both levers at once.** `mechanical` uses the small model but
  at `medium` effort, not `low` — a small model at minimal effort is where
  reliability quietly drops on exactly the work you wanted cheap-but-correct.
- **sol is strong at low effort** (per its own NUX copy): reserve `xhigh`/`max`/
  `ultra` for genuinely hard problems; `hard` defaults to `high`.
- **Fallback = availability, not cost.** A class's fallback list exists so an
  *unavailable* primary can still run; every entry must be a slug the account can
  execute. It is not a cost-optimization knob.

### 3.3 Loader validation (`src/config/modelPolicy.ts`)

Add one validation pass with an explicit rule for the missing-`efforts` case
(resolving the tension the reviewer flagged between this section and §8):

- If `models[model].efforts` **is present**, assert `effort ∈ efforts` for every
  class and review type. A violation **throws at load** with a precise message
  (`class "hard" uses effort "ultra" not supported by model "gpt-5.6-luna"`).
- If `models[model].efforts` **is absent** (an older local override), validation
  is **skipped** for that model — the loader does **not** throw. Instead doctor's
  `models` row emits a `WARN` telling the user to declare `efforts` (or run
  `refresh-models`, §4.3, which fills them from the live catalog).

So: declared-and-wrong is a hard load error; undeclared is a soft doctor warning.
This keeps the shipped template strict while never breaking an in-place upgrade.
`resolve()` is otherwise unchanged (still returns `{ chain, effort, timeoutMs }`).

## 4. Doctor hardening & real `refresh-models`

### 4.1 Model catalog source of truth

Introduce a small IO unit `src/exec/modelCatalog.ts` that runs
`codex debug models` (via the existing injectable `Runner`) and parses the JSON
into `{ slug, efforts }[]`. Pure parsing is split from the spawn so it is unit
tested against a captured fixture.

### 4.2 New doctor rows

- **`models`** — every slug referenced by the active `model-policy.toml`
  (classes + review) is present in the catalog **and** the configured effort is
  in that slug's supported set. Missing slug → `MISSING` with remediation
  ("run `codex update`; then confirm the slug via `codex debug models`").
  Unsupported effort → `WARN`.
- **`cli-version`** — compare the running CLI version against the latest the
  catalog/`codex update --check` reports; stale → `WARN` with `codex update`
  (this is precisely the trap that hid 5.6 today).

Both degrade gracefully (a probe failure is `WARN`, never a hard crash of doctor)
— consistent with the existing `probeGlobalCodexRouting` philosophy.

**Injection seam (named so the plan doesn't rediscover it):** doctor's
dependencies are the `DoctorDeps` interface in `src/doctor.ts`, assembled by
`buildDoctorDeps` (`cli.ts:110`). The new rows add one dep — a
`readModelCatalog: () => Promise<CatalogEntry[]>` (backed by `modelCatalog.ts`,
§4.1) — wired exactly like the existing `checkProviderRouting` dep. `runDoctor`
gains the catalog dep and the two new rows; nothing else in the doctor pipeline
changes. The `models` row reads the *active* policy (same `resolvePolicyPath`
lookup `buildDoctorDeps` already uses for `policyExists`) to know which slugs to
check. Hidden catalog entries (e.g. `codex-auto-review`, `visibility: "hide"`)
are ignored unless a policy actually references them.

### 4.3 `refresh-models` implemented (replaces the stub)

Today `handleRefreshModels` is a no-op waiting on `OPENAI_API_KEY` + `/v1/models`.
Repoint it at `codex debug models` (no API key, reflects the *account's* real
catalog) and have it **print a proposed `model-policy.toml` diff**: unknown slugs
to remove, new siblings available, effort mismatches. It writes nothing on its
own — it proposes; the human applies. This finally makes the command useful and
closes the "keep the template's IDs current" loop the template only describes in
prose.

**Exit-code contract (pinned):** `refresh-models` exits `0` whether or not it
finds anything to propose (it is advisory, like the stub it replaces — a clean
"no changes needed" is success). It exits non-zero **only** when the catalog
itself cannot be read (e.g. `codex debug models` fails or returns unparseable
JSON), so a broken CLI is visible. The old `OPENAI_API_KEY` branch is removed.

## 5. Read-only review subsystem ("judge" mode)

### 5.1 The distinct safety contract

Reviews read; they never write. So the *execute* contract does not map onto them
and must not be reused wholesale:

| guard                   | execute            | review (judge)                         |
| ----------------------- | ------------------ | -------------------------------------- |
| whitelist (non-empty)   | **required**       | **absent** (nothing is written)        |
| auto-revert off-list    | yes                | n/a (read-only sandbox forbids writes) |
| clean-tree preflight    | required (`ask`)   | **skipped** — reviewing a dirty/in-progress tree is the point |
| protected-path deny-list| enforced           | moot for writes; still recorded        |
| sandbox                 | workspace-write+   | **`read-only`**                        |
| `approval_policy=never` | invariant          | invariant                              |
| ledger row              | yes                | yes (`mode: "review"`, reviewType, model, effort) |

Adding a `read-only` sandbox level is a **multi-site** change (the reviewer
correctly flagged that "single site" was wrong): the `SandboxLevel` union and the
`SANDBOX_LEVELS` const array (`types.ts:19-21`) both gain the member, the
`sandboxFlags` switch (`codexArgs.ts:23-46`, exhaustive with no `default` arm)
gains its case `['--sandbox', 'read-only']`, and `validateDelegationSpec` (which
checks membership) picks it up automatically via `SANDBOX_LEVELS`. What *does*
stay in one place is the **flag mapping** — `codexArgs.ts` remains the only spot
that turns a level into concrete `--sandbox` flags. `codex exec` accepts
`read-only` as a first-class `-s` value (verified in `codex exec --help`:
`[possible values: read-only, workspace-write, danger-full-access]`).

### 5.2 `ReviewSpec` and policy `[review]` section

```ts
export type ReviewType = 'code-review' | 'audit' | 'plan-review';

export interface ReviewSpec {
  readonly reviewId: string;
  readonly repoPath: string;          // absolute
  readonly reviewType: ReviewType;
  readonly target: string;            // git ref/range | path/area | plan-file path
  readonly focus?: string;            // e.g. "security" for audit
  readonly model?: string;            // override; else policy [review] default
  readonly effort?: Effort;           // override; else policy [review] default
  readonly auth?: AuthMode;           // reuse existing rotate/native
}
```

Policy defaults (same TOML — single source of truth):

| reviewType    | model           | effort | fallback        |
| ------------- | --------------- | ------ | --------------- |
| `code-review` | `gpt-5.6-terra` | high   | `gpt-5.5`       |
| `audit`       | `gpt-5.6-sol`   | xhigh  | `gpt-5.6-terra` |
| `plan-review` | `gpt-5.6-sol`   | high   | `gpt-5.6-terra` |

`code-review` defaults to terra (routine diffs are frequent → cost-aware), with
sol as a conscious on-demand override for critical diffs. `audit`/`plan-review`
default to sol (infrequent, high-value).

### 5.3 Hybrid engine

- **`code-review` → wraps the native `codex review`.** Build its argument array
  (`review`, plus `--base <branch>` / `--uncommitted` / `--commit <sha>` derived
  from `target`, custom instructions via stdin). Best-in-class diff handling,
  less to maintain. Still routed through the shared `Runner`, fallback ladder,
  auth, and ledger.
  - **Model override — `-c model=`, not `-m`.** Verified in `codex review
    --help`: the subcommand exposes **no `-m/--model` flag** — its options are
    `-c/--config`, `--strict-config`, `--enable`/`--disable`, `--uncommitted`,
    `--base`, `--commit`, `--title`. The model is therefore set via the generic
    config override `-c model="gpt-5.6-terra"` (the same `-c key=value` mechanism
    `codex exec` uses; the help's own example is `-c model="o3"`). Left
    unoverridden, native review uses its dedicated `codex-auto-review` model.
    Whether `-c model=` actually *retargets* the review (vs. the backend forcing
    `codex-auto-review`) is not yet runtime-confirmed → tracked in §9.2. If the
    override is ignored at runtime, `code-review` falls back to the same custom
    read-only exec path as audit/plan-review (below), with the diff injected into
    the prompt — so the design does not collapse either way.
- **`audit` / `plan-review` → custom read-only exec, reusing `buildCodexArgs`.**
  These go through the **existing** `codex exec` path with `sandboxLevel:
  'read-only'` — i.e. `reviewController` builds a `CodexInvocation` and calls the
  same `buildCodexArgs` (§5.1), so the `--sandbox`/effort/`approval_policy` flags
  and the model (`-m`) come from the one mapping site, not a reimplementation.
  The only review-specific pure unit is the **prompt**: `src/reviewPromptBuilder.ts`
  (twin of `promptBuilder.ts`) produces a purpose-built prompt per reviewType
  (native review only understands diffs, not "audit this area" or "review this
  written plan"), imposing a structured findings format:
  `severity | file:line | claim | why`.

This resolves the earlier ambiguity about `reviewArgs.ts`: it builds **only** the
native-`codex review` argument array (a different command from `codex exec`); the
read-only exec path reuses `buildCodexArgs` unchanged. Both engines deliver the
prompt via **stdin (`-`)**, matching the Windows-safe discipline documented in
`codexArgs.ts` (never pass a multiline prompt as argv).

### 5.4 Reuse map

Reuses without change: `Executor`, `MultiAuth`, `classifyFailure`, `fallback`,
`Ledger`, `run`, and `buildCodexArgs` (for the read-only exec path). New units:

- `src/reviewPromptBuilder.ts` (pure) — the per-reviewType prompt.
- `src/exec/reviewArgs.ts` (pure) — builds **only** the native-`codex review`
  argument array (`--base`/`--uncommitted`/`--commit`, `-c model=`).
- `src/reviewController.ts` (IO) — orchestration mirroring `controller.ts` but
  with the **reduced review preflight** (git-repo check for `code-review`; none
  for `plan-review`, whose target is a text file) and **no verifier/snapshot**
  (nothing is written, so there is nothing to verify or revert).

**CLI seam (named):** `CliHandlers` and the `dispatch` switch (`cli.ts:45-78`)
currently hard-code `doctor`/`delegate`/`refresh-models`. The plan extends
`CliHandlers` with `review`/`audit`/`plan-review`, adds their `dispatch` cases
(each reading a `ReviewSpec` JSON file, mirroring how `delegate` reads a
`DelegationSpec`), and updates the usage/error strings. A shared
`validateReviewSpec` (twin of `validateDelegationSpec` in `preflight.ts`) rejects
a malformed spec before any spawn.

### 5.5 Advisory contract (output handling)

The review command prints a JSON outcome `{ status, findings, model, effort }`.
Findings are **advisory**. The skill (§6) instructs Claude to: read findings,
**verify the highest-severity ones against the actual code/plan**, then decide —
never apply blindly, never dismiss blindly. This is how "Codex may be more precise
than Claude" coexists with "Claude keeps a verification loop and stays
accountable for what lands."

## 6. Capability map — `SKILL.md` rewrite (primary deliverable)

The most valuable artifact is not the new subcommand code (mostly reuse) — it is
Claude's **awareness** of the whole arsenal at planning time. Rewrite
`skills/codex-delegate/SKILL.md` so it teaches:

1. **Two modes.** *Execute* (writes, whitelist-guarded, clean-tree required) vs
   *Judge* (read-only, advisory, dirty tree fine). When to reach for each.
2. **A decision table** task → mode → model → effort → cost, replacing the old
   "you only pick effort" framing. Concrete target structure the plan must
   produce (not just themes):

   | I want to…                        | mode    | subcommand    | model         | effort | cost |
   | --------------------------------- | ------- | ------------- | ------------- | ------ | ---- |
   | rename/apply-diff/boilerplate     | execute | `delegate`    | `gpt-5.6-luna`  | medium | $    |
   | build a feature / guided refactor | execute | `delegate`    | `gpt-5.6-terra` | medium | $$   |
   | crack a hard, specifiable bug     | execute | `delegate`    | `gpt-5.6-sol`   | high   | $$$  |
   | sanity-check a routine diff       | judge   | `review`      | `gpt-5.6-terra` | high   | $$   |
   | critical diff / security-sensitive| judge   | `review`      | `gpt-5.6-sol`   | high   | $$$  |
   | audit an area for issues          | judge   | `audit`       | `gpt-5.6-sol`   | xhigh  | $$$$ |
   | second-opinion a plan before work | judge   | `plan-review` | `gpt-5.6-sol`   | high   | $$$  |

   The skill states these are policy *defaults* (from `model-policy.toml`) and
   that Claude may override model/effort on a spec when a specific job justifies
   it — up or down.
3. **The Sol clause:** "sol can exceed your precision on judgment — treat its
   review as a genuine second opinion, but verify its findings against the code
   before acting on them. You remain accountable for what you apply."
4. **Cost discipline (autonomous, no hard gate):** sol is the most expensive; use
   it where correctness pays for it (hard execution; audit/plan-review; critical
   code-review). Don't spend sol on trivial diffs — terra is the routine default.
5. **The planning hook:** when a plan step says "delegate to Codex here," Claude
   should already know whether that means execute or judge, at what model/effort,
   and what it will do with the result.

The old contradictory guidance ("do NOT pick a different model for easy tasks")
is removed; the "When NOT to delegate" section is retained and extended to the
judge mode (don't ask for a review you won't read/verify).

## 7. Testing strategy

Preserve the project's pure/IO split and TDD discipline:

- **Pure, exhaustively tested:** extended `Effort`; `(model, effort)` catalog
  validation in `modelPolicy`; `reviewPromptBuilder` (per reviewType, focus,
  structured-format assertions, verbatim target embedding); `reviewArgs`
  (native-review flag derivation from `target`; read-only exec arg set; asserts
  **never** `workspace-write`/`danger-full-access`/`--dangerously-*` for a
  review; prompt via `-`); model-catalog JSON parser against a captured fixture.
- **IO, mock-injected:** `reviewController` (advisory outcome shape; reduced
  preflight — git-repo check for code-review, none for plan-review; ledger row
  written); doctor `models`/`cli-version` rows against fake catalog/runner;
  `refresh-models` diff proposal against a fixture catalog.
- **Regression:** existing execute-path tests must stay green untouched
  (contract unchanged).

## 8. Migration & compatibility

- Existing `.codex-delegate.local/model-policy.toml` overrides without an
  `efforts` field on models: the loader **skips** `(model, effort)` validation
  for those models (no throw) and doctor `models` emits a `WARN` until they are
  declared — the exact rule fixed in §3.3.
- Overrides without a `[review]` section: the review subcommands fall back to the
  **built-in defaults hard-coded in `reviewController`** (matching the §5.2
  table), so review works on a pre-upgrade policy without edits. Document the
  upgrade path in `CHANGELOG.md`.
- The `read-only` sandbox and 5.6 slugs require CLI ≥ 0.144.1 — doctor's
  `cli-version` row surfaces this.

## 9. Open items to resolve during planning

1. **Runtime ChatGPT-auth availability** of `gpt-5.6-{sol,terra,luna}` — confirm
   via the doctor probe / one cheap `codex exec -m gpt-5.6-sol` before making the
   new template the shipped default. If any slug is API-only on this account,
   keep 5.5/5.4 as the committed default and gate the 5.6 slugs behind the probe.
2. **Native `codex review`: output shape + model override.** Two things to
   confirm from one captured real run: (a) the stdout format, to pin the
   `reviewController` parser (severity/file/line fields); and (b) whether
   `-c model="…"` actually retargets the review model or the backend forces
   `codex-auto-review` (per §5.3 — determines whether the terra-default /
   sol-on-demand knob works for `code-review`, or whether `code-review` must fall
   back to the custom read-only exec path).
3. **`ultra` semantics** ("maximum reasoning with automatic task delegation") —
   decide whether to expose it at all in v1 or reserve it (sol/terra only; luna
   lacks it). Leaning: allow in policy, never a v1 default.

## 10. Sequencing (plannable units)

The design is two loosely-coupled subsystems sharing only `modelCatalog.ts`. They
should be planned and can ship as **two phases**:

1. **Phase A — models + doctor.** `Effort`/`ModelTier`/`ModelEntry` changes,
   template rewrite, loader validation (§3), `modelCatalog.ts`, doctor `models`/
   `cli-version` rows, real `refresh-models` (§4). This phase also *provides the
   probe* that §9.1 needs to confirm 5.6 runtime availability — so it must land
   first regardless.
2. **Phase B — review subsystem + capability map.** `read-only` sandbox,
   `ReviewSpec`, `reviewPromptBuilder`, `reviewArgs`, `reviewController`, the
   three subcommands (§5), and the `SKILL.md` rewrite (§6).

Phase B's `SKILL.md` decision table depends on Phase A's finalized model mapping,
so A→B is the natural order. Each phase is independently testable and shippable;
`writing-plans` may produce one plan with two chunks or two plans.
